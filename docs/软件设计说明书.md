# 项目阶段汇总服务 软件设计说明书

## 1. 文档目的

本文档基于当前代码版本，说明“项目阶段汇总服务”的系统结构、核心模块、数据流、权限模型、定时任务、部署方式与主要设计取舍，用于后续运维部署、功能扩展和代码维护。

## 2. 系统概述

项目阶段汇总服务是一个独立部署的 ASP.NET Core Minimal API Web 服务，面向项目阶段汇总、看板展示、人员统计、项目分配与通知场景。

系统当前版本已经从“直接查询多源数据库”的模式，演进为“中心表驱动”的模式：

- 源数据库包括 SQL Server 与 MySQL
- 主应用负责配置、调度、查询、权限控制、导出、看板展示
- 中心库使用 SQL Server
- 项目阶段、人数等汇总结果最终统一写入中心表
- 页面查询、导出、看板均优先读取中心表

此外，系统支持两类采集方式：

- 直连采集：主应用直接访问源库
- Agent 采集：部署在业务服务器上的轻量 Agent 本地访问源库，再写入中心库

## 3. 设计目标

当前版本的设计目标如下：

- 支持多源异构数据库接入
- 支持在 Windows 环境稳定部署
- 通过中心表提升查询性能与稳定性
- 通过角色和项目负责人实现数据权限控制
- 通过本地缓存降低页面查询对源库的依赖
- 支持定时刷新、定时人数更新、钉钉日报通知
- 前端以看板为核心，兼顾项目配置和服务器资源观察

## 4. 总体架构

### 4.1 逻辑分层

系统逻辑分为 5 层：

1. 前端展示层  
   单页页面，位于 `wwwroot/index.html`，负责登录、筛选、看板展示、配置弹窗、统计摘要和交互。

2. Web API 层  
   使用 Minimal API，在 `Program.cs` 中定义所有接口，负责认证、授权、参数接收、业务编排和结果输出。

3. 业务服务层  
   位于 `Services/` 目录，负责查询、刷新、中心表同步、缓存、用户、项目元数据、钉钉通知等逻辑。

4. 存储层  
   包括：
   - 本地 SQLite 缓存
   - 本地文件配置
   - 中心 SQL Server 汇总表
   - 中心 SQL Server 项目元数据表

5. Agent 层  
   部署在业务服务器侧的轻量服务，用于本地读取源库并写入中心库：
   - Debian/Python 版：`tools/stage_agent/stage_agent.py`
   - Windows/.NET 版：`tools/stage_agent_win/`

### 4.2 典型运行模式

#### 模式 A：纯直连

- 主应用读取服务器配置
- 直接访问 SQL Server / MySQL 源库
- 查询结果先写入本地 SQLite 缓存
- 若启用中心库，再同步写入中心表

#### 模式 B：Agent + 中心表

- 主应用不直接访问部分业务库
- 主应用调用业务服务器上的 Agent `/sync`
- Agent 在本地读取源库并写入中心表
- 主应用页面查询仅读中心表

该模式适用于 VPN、内网隔离、业务库只能本机访问等场景。

### 5.7 刷新服务

文件：

- `/Users/dalezhao/python_learning/project_stage_service/Services/ProjectStageRefreshService.cs`

职责：

- 手工或定时触发阶段刷新
- 分流 Agent 服务器与直连服务器
- 聚合刷新结果
- 写本地缓存
- 在启用中心库时同步写中心表

刷新流程：

1. 读取启用的服务器配置
2. 将服务器分为 Agent 组和直连组
3. Agent 组并行调用 `/sync`
4. 直连组由查询服务执行全量阶段扫描
5. 汇总结果保存到 SQLite 缓存
6. 若中心库启用，同步写入中心表

### 5.8 人数刷新服务

文件：

- `/Users/dalezhao/python_learning/project_stage_service/Services/ProjectStageCountRefreshService.cs`

职责：

- 基于缓存中的项目目标集合刷新人数
- 按服务器串行执行
- 对每台服务器先更新报名人数，再更新准考证人数
- 刷新完成后回写缓存
- 若中心库启用，再同步中心表

设计特点：

- 避免全量重扫业务阶段
- 只对缓存中存在的目标项目执行人数统计
- 有独立互斥锁，防止并发重复执行

## 8. 权限设计

### 8.1 认证方式

- Cookie 认证
- API 未登录返回 401
- 首次登录强制改密时，大部分 API 返回 403

### 8.2 授权策略

- `AdminOnly`
  - 要求 `role=admin`

- `InternalOrAbove`
  - 要求 `role=admin/internal`

### 8.3 角色权限矩阵

#### admin

- 用户管理
- 中心库配置
- 调度配置
- 钉钉主通知
- 查看所有项目
- 管理服务器配置
- 维护项目负责人和应用服务器

#### internal

- 查看所有项目
- 管理服务器配置
- 测试连接
- 维护项目负责人和应用服务器
- 管理应用服务器候选项

#### external

- 只能查看分配给自己的项目
- 只能修改自己项目的应用服务器分配
- 不可管理服务器
- 是否可触发刷新，由管理员单独配置

### 8.4 数据级权限

外部用户访问查询、阶段、导出等接口时，会在服务端按 `project_metadata.maintainer` 过滤，只保留：

- `server_name + exam_code` 属于当前用户的项目

该过滤在服务端完成，不依赖前端隐藏。

## 10. 关键业务流程

### 10.1 登录流程

1. 首次访问，若未初始化账号，则引导设置管理员账号
2. 用户登录后建立 Cookie 会话
3. 若为默认密码用户，强制修改密码

### 10.2 数据刷新流程

1. 用户或调度触发刷新
2. 读取服务器配置
3. Agent 服务器并行调用 Agent
4. 非 Agent 服务器由主应用直连扫描
5. 汇总后写入 SQLite 缓存
6. 若启用中心库，再同步写入 `project_stage_summary`

### 10.3 页面查询流程

1. 前端构造 `ProjectStageQueryRequest`
2. API 从中心表读取汇总数据
3. 若当前用户是 external，则按负责人过滤
4. 返回看板数据和摘要统计

### 10.4 人数刷新流程

1. 从缓存中读取项目目标
2. 逐服务器串行执行
3. 对每台服务器先刷新报名人数，再刷新准考证人数
4. 回写缓存
5. 若启用中心库，再同步中心表

### 10.5 钉钉日报流程

1. 定时读取当天开始阶段
2. 汇总总体统计
3. 发送管理员主通知
4. 按负责人分发个人通知

## 13. 异常与容错设计

### 13.1 并发保护

- 阶段刷新有互斥锁
- 人数刷新有互斥锁
- 避免调度和手工触发同时执行

### 13.2 失败隔离

- Agent 失败不会阻塞其他 Agent
- 某台服务器人数刷新失败不影响前面已完成服务器
- 中心表启用时，查询侧不再依赖源库实时可用

### 13.3 数据一致性

- 本地缓存和中心表采用快照式刷新
- 中心表写入前按源分组清理
- 写入时按唯一键去重
